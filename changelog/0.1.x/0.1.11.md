# 0.1.11 — 2026-04-24

Closes [#1](https://github.com/cyanheads/arxiv-mcp-server/issues/1) and [#2](https://github.com/cyanheads/arxiv-mcp-server/issues/2).

## Fixed

- **`arxiv_search` / `arxiv_get_metadata`** — `fetchApi` no longer retries permanent 4xx responses from the arXiv API as transient. Previously, bad input (e.g. `max_results=1.5`, malformed queries) returned HTTP 400 with `application/atom+xml` content-type and was wrapped in `TransientError`, triggering 3 retries with exponential backoff — adding ~7s of caller-visible latency per failed call and 4x'ing upstream request volume on caller-side bad input. Now only 429 and 5xx throw `TransientError`; other 4xx throw `validationError` and surface immediately. ([#1](https://github.com/cyanheads/arxiv-mcp-server/issues/1))
- **`arxiv_search` input schema** — inputs that were silently forwarded upstream and rejected with generic HTTP 400/500 errors are now caught by Zod at the tool boundary: `query` is `.trim().min(1).max(1000)` and rejects C0 control characters; `max_results` is `.int()` (1–50); `start` is `.int().min(0).max(10_000)` to stay inside arXiv's practical pagination window. ([#2](https://github.com/cyanheads/arxiv-mcp-server/issues/2))
- **`arxiv_read_paper.max_characters`** — now `.int().min(1)` so fractional and non-positive character budgets are rejected at the Zod layer.
